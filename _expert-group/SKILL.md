# 权威合议组（强制输出版）

## 1. 核心指令

用户输入为「议题」，需完成完整合议流程并产出成果，禁止跳过阶段直接输出结论。

## 2. 定义与约束

### 2.1 核心定义

1. **终极用户**：对成果价值有评判权的主体（需求方、服务对象、决策者、监管方、客观标准）
2. **议题画像**：议题的结构化解构成果，作为研讨基石
3. **盲区维度**：用户大概率未曾考虑的认知视角

### 2.2 执行约束

1. **长内容控制**：输出接近上限时，在逻辑节点暂停并提示「继续？」
2. **信息补充**：必要时可联网查询时效性素材
3. **双轨输出**：每阶段内容须明文显示并同步保存至文件

### 2.3 路径约定

全流程涉及三类互不相同的目录，严禁混用：

| 标识 | 名称 | 基准 | 用途 | 读写 |
|------|------|------|------|------|
| **SKILL_DIR** | Skill 目录 | 本 SKILL.md 所在目录 | 静态资源（含 `experts/` 专家定义） | 只读 |
| **PROJECT_ROOT** | 运行时项目根目录 | 调用方当前会话的项目工作区根 | 挂载合议输出目录 | 可写 |
| **SESSION_DIR** | 合议输出目录 | `{PROJECT_ROOT}/logs/_expert-group/{unix_timestamp}-{议题简称}/` | 本次合议全部产出文件 | 可写 |

- `experts/` 引用一律指 `{SKILL_DIR}/experts/`
- `proceedings.md`、`result.md`、`blindspot.md`、`progress.md`、各专家发言文件一律落在 `{SESSION_DIR}`
- 后文出现「合议输出目录」「本目录」「当前会话目录」均指 **SESSION_DIR**

### 3.2 加载规则

1. **必选角色**：会议主席、反对派必须加载
2. **按需匹配**：根据议题画像识别关联维度，加载对应专家
3. **多实例配置**：同类专家可配置多人覆盖不同子领域
4. **动态补位**：根据识别的盲区维度，加载补位专家填补缺口

## 4. 工作流程

### 4.1 解构议题

分析输入，建立「议题画像」。

**4.1.1 基础解构**

- **基本面**：议题类别、意图、约束、背景、预期、交付对象
- **深层要素**：利益诉求、担忧、痛点、潜在期望
- **终极用户**：列举所有终极用户及其核心关切

**4.1.2 认知边界测绘**

- **已知边界**：用户展现的知识范围、假设前提、思维框架
- **信息来源推断**：可能接触的信息渠道及其局限
- **思维定式识别**：隐含的因果假设、价值排序、路径依赖

**4.1.3 盲区预判**

- **结构性盲区**：因位置、角色、专业导致的系统性忽略
- **时间性盲区**：聚焦当下而忽略的历史规律与未来演化
- **尺度性盲区**：局限于单一尺度而错失的跨尺度洞察
- **反事实盲区**：未曾设想的替代现实

**4.1.4 破局关键**

- **隐性杠杆**：撬动全局的关键变量
- **瓶颈命门**：制约效能的核心卡点
- **分析路径**：需引入的专业领域与认知层级

**输出**：议题画像 → 暂停，提示用户输入「修改建议」→ 整合后进入下一阶段

### 4.2 构建合议组

根据议题画像动态加载专家定义，匹配权威阵容。

**选配原则**

1. **多元覆盖**：同一类别可配置多人覆盖不同子领域
2. **不可替代**：每位专家须有独特贡献点
3. **规模弹性**：简单议题 9–15 人，复杂议题 15–21 人
4. **盲区优先**：优先覆盖核心领域与盲区维度
5. **规模底线**：人数不得低于下限，选配后须核对达标

**输出**：权威列表（序号、姓名、角色、专业身份、入选理由、预期贡献的盲区维度）+ 人数统计确认

### 4.3 深度研讨

**4.3.1 定义成果（主席）**

主席发布最终交付成果定义：

- **成果蓝图**：载体、规范、对象、场景、价值
- **评价体系**：100 分制；至少覆盖四类维度——形态适配、受众契合、效能价值、认知增量；另必设**「思维品质」维度（占比 ≥ 30 分）**，细分指标为见体 / 存异 / 见网 / 动态 / 溯源（对应 §4.4.2 五字约束）
- **盲区覆盖目标**：至少包含 5 个用户未曾考虑的有效视角

**输出**：成果蓝图、评价体系 → 暂停，提示「继续？」

**4.3.2 独立陈述**

每位专家紧扣议题画像发表独立见解。

**发言规范**

1. **维度纵深**：聚焦领域独有洞察、框架、方法论
2. **盲区贡献**：明确指出用户大概率未曾考虑的内容
3. **有据可依**：以事实、理论、数据、案例支撑，标注置信度
4. **实操路径**：论证充分的实施路径
5. **充分展开**：详尽阐释
6. **风格拟真**：契合角色人设的风格、术语、思维

**发言顺序**

1. 情报权威（信息校准）
2. 专业权威/实践权威（领域深挖）
3. 经济学家（资源配置与激励）
4. 第一性原理权威/规律权威（本质溯源）
5. 复杂系统洞察权威（系统关联）
6. 盲区猎手/黑天鹅权威/反常识权威（边界拓展）
7. 高阶上位者（降维视角）
8. 补位权威（盲区填补）
9. 反对派（系统审视，最后发言）

**执行方式（逐人独立文件）**

每位专家的发言必须**逐人完成**，严格按以下循环执行，禁止在一次输出中合并多位专家：

1. 在对话中明文输出当前专家的完整独立发言
2. 立即将该发言写入独立文件：`4.3.2-{序号}-{姓名}-{角色简称}.md`（如 `4.3.2-01-王明远-情报权威.md`）
3. 更新 `progress.md`，勾选该专家条目
4. 确认文件已保存后，再进入下一位专家，重复上述步骤

> **强制约束**：每位专家必须拥有独立文件，一人一文件，先输出再写文件，写完再启动下一位。此机制确保每人的观点独立生成、不互相污染。

**长内容处理**：输出接近上限时，在当前专家发言及文件保存完毕后暂停提示「继续？」，下次从下一位专家继续

### 4.4 成果交付

**4.4.1 议题综述（主席）**

本阶段是研讨沉淀，非成果交付。仅整合、不定稿，输出物作为 4.4.2 的原料。

- **核心洞察**：整合见解，提炼本质判断
- **认知突破点**：3–5 个最具价值的新视角
- **分歧矩阵**：显式保留专家间的主要对立与开放张力（作为 4.4.2 的凝结燃料），不得抹平
- **盲区清单**：汇总识别的盲区（编号、描述、贡献者、优先级）
- **遗留盲区**：标注未覆盖的潜在盲区

**输出**：议题综述（注：决策建议、风险清单、行动方案、合规自检等交付级要件一律下放到 4.4.2）

**4.4.2 最终成果**

承接 4.4.1 的沉淀材料（核心洞察、分歧矩阵、盲区清单），此处完成最终凝结。成果面向终极用户，可含**决策建议、风险对冲、行动方案**等要件，但具体形态依议题特质动态决定（设计稿、代码、方案、答案、图谱、剧本、对话录……皆可）。无论承载何种形态，输出本体都应是**「凝结的思考」**，而非**「抛光的结论」**。

**思维品质约束**（内化于成果肌理，不以清单形式机械堆砌，而是以形态本身允许的语言表达出来——代码可以借注释与命名，设计稿可以借备选与图注，答案可以借转折与旁白）：

- **见体**：任何主张背后至少含一次「何以如此」的追问，让表象透出本质；
- **存异**：研讨中已浮现的张力、分歧、博弈、权衡，必须以形态允许的方式留痕，不得为顺滑而抹平；
- **见网**：结论不悬空孤立，应显露其与上下文、周边要素、反馈回路的联结；
- **动态**：凡涉方案、主张、判断者，须隐含其适用边界、失效信号、演化路径，哪怕只一句旁注；
- **溯源**：关键判断可追溯至研讨中的专家观点或事实基座，不作凭空跃迁。

**判别线**：若成果读起来像「结论的复述」而非「思考的凝结」，即退回重做。哲学意义上，成果呈现的不是「是什么」，而是「何以如此、何以不如此、何时不如此」。

**合规自检**：输出前对照 4.3.1 评价体系逐项自评打分（含「思维品质」维度），不合格须退回 4.4.1 重新切片。

**输出**：最终成果 + 自检分表

**4.4.3 认知拓展备忘**

提炼后续思考方向：

- **值得追问的问题**：5–10 个深挖问题
- **推荐探索方向**：相关但未深入的领域

**输出**：认知拓展备忘

## 5. 输出与存储规则

### 5.1 基本原则

1. **明文输出**：各阶段内容须在对话中以纯文本形式完整显示
2. **同步存档**：无论被调用方式如何，须将内容写入文件
3. **格式清晰**：使用清晰的标题层级和分隔线

### 5.2 文件存储规则

**目录命名**：SESSION_DIR = `{PROJECT_ROOT}/logs/_expert-group/{unix_timestamp}-议题简称/`

- 时间锚点：`unix_timestamp` 取 §4.1 开始时刻的 Unix 秒级时间戳，全程保持不变
- 同名则追加序号（如 `-2`）
- 以下所有文件路径均相对 SESSION_DIR

**阶段 4.1：解构议题**

1. 明文输出：完整议题画像
2. 文件写入：`proceedings.md` —— 原始议题输入 + 4.1 议题画像 + 用户修改建议及整合结果（如有）
3. 完成标签：`[4.1 完成 | 时间戳]`
4. 确认回执：向用户显示文件路径

**阶段 4.2：构建合议组**

1. 明文输出：完整权威列表
2. 文件追写：`proceedings.md` —— 4.2 权威列表
3. 完成标签：`[4.2 完成 | 人数统计：X人]`
4. 确认回执：向用户显示文件路径

**阶段 4.3.1：定义成果**

1. 明文输出：成果蓝图 + 评价体系 + 盲区覆盖目标
2. 文件追写：`proceedings.md` —— 4.3.1 成果蓝图与评价体系
3. 完成标签：`[4.3.1 完成]`

**阶段 4.3.2：独立陈述**

1. 明文输出：逐人完整发言（一人一次，不得合并）
2. 独立存档：每位专家写入独立文件 `4.3.2-{序号}-{姓名}-{角色简称}.md`
3. 串行铁律：每人按「明文 → 独立文件 → 更新 `progress.md` → 下一位」的顺序执行
4. 汇总索引：全部专家发言完毕后，`proceedings.md` 追加 4.3.2 索引（专家姓名 → 文件路径）
5. 完成标签：`[4.3 完成]`
6. 中断机制：输出接近上限时，在当前专家发言及文件保存完毕后明文提示「**暂停**已保存进度，输入**继续**」

**阶段 4.4：成果交付**

1. 明文输出：议题综述 + 最终成果 + 自检分表 + 认知拓展备忘
2. 文件追写：`proceedings.md` —— 4.4 完整内容（含自检分表）
3. 文件新建：`result.md` —— 最终成果（含自检分表）
4. 文件新建：`blindspot.md` —— 盲区清单 + 认知拓展备忘
5. 完成标签：`[流程完成]`
6. 完成回执：明文显示「合议流程完成」 + 所有文件路径列表

### 5.3 进度档案（progress.md）

在工作目录下维护 `progress.md` 文件，作为流程断点恢复的唯一依据。

**格式规范**

```markdown
# 合议进度

- 议题：{议题简称}
- 目录：{工作目录路径}
- 启动时间：{unix_timestamp}

## 阶段进度

- [x] 4.1 解构议题
- [x] 4.2 构建合议组（{X}人）
- [ ] 4.3 深度研讨
  - [x] 4.3.1 定义成果
  - [ ] 4.3.2 独立陈述
    - [x] 4.3.2-01-王明远-情报权威
    - [x] 4.3.2-02-张建国-领域专家
    - [ ] 4.3.2-03-李芳-经济学家  ← 当前
    - [ ] 4.3.2-04-...
- [ ] 4.4 成果交付
```

**更新时机**

1. **4.1 开始**：创建 `progress.md`，记录议题与目录信息
2. **每个阶段完成**：勾选对应阶段
3. **每位专家发言完成**：勾选对应专家条目
4. **4.4 全部完成**：所有条目勾选，追加 `完成时间：{unix_timestamp}`

### 5.4 启动时断点检测

每次启动合议流程时，**优先扫描** `{PROJECT_ROOT}/logs/_expert-group/` 下所有子目录的 `progress.md`：

- **存在一个或多个「未全部勾选」的 SESSION_DIR**：向用户列出候选（目录名 + 议题简称 + 当前停留阶段），询问「检测到未完成的合议进度，继续请选择目录编号；开启新议题请输入 `新建`」→ 用户选定后，将该目录设为当前 SESSION_DIR 从断点恢复
- **所有子目录均已勾选完毕**或 `logs/_expert-group/` 不存在：正常启动新流程，按 §5.2 阶段 4.1 创建新 SESSION_DIR
- **`progress.md` 缺失或损坏**：视该目录为废弃，跳过

### 5.5 执行检查清单

每阶段自检：

- [ ] 内容已在对话中明文显示
- [ ] 已写入文件
- [ ] 已更新 `progress.md`
- [ ] 已向用户确认文件保存路径
- [ ] 格式符合规范
