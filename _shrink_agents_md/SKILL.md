# 工作流程

## 1. 备份原文件

1.1 在执行任何修改前，备份原始文件
1.2 备份命名：`{原文件名}.{unix时间戳}.bak`（如 `AGENTS.md.1711089600.bak`）
1.3 备份文件与原文件位于同一目录

## 2. 分析原文件

2.1 通读原文件，识别所有指令段落
2.2 按职责归类，标记每段属于哪个层级（见下方分层原则）
2.3 识别段落间的依赖关系和加载优先级

## 3. 设计拆分方案

3.1 根据分层原则确定模块目录结构（目录名、文件名根据实际内容动态命名）
3.2 确定入口文件的路由逻辑（哪些模块始终加载，哪些按需引用）
3.3 处理跨模块依赖：提取共享定义到公共模块，或在入口中声明

## 4. 执行拆分

4.1 逐模块提取内容，保持原始语义不变
4.2 每个模块文件应自包含、职责单一
4.3 生成入口文件：只包含角色一句话定义、模块加载指令、全局约束（如有）

## 5. 验证

5.1 检查：原文件所有指令是否已被覆盖，无遗漏
5.2 检查：入口文件是否足够精简，无冗余内容
5.3 检查：模块间无循环依赖、无重复定义

# 分层原则

不预设固定层级模板。基于对原文件的语义理解和逻辑关系，动态决定分层方案：

## 分层思路

1. **识别语义边界**：找到内容在「讲什么」上的自然断裂点——话题切换、职责转移、抽象层次变化都是潜在切割位
2. **判断内聚度**：同一模块内的段落应围绕单一关注点，互相之间高度相关；若移除其中一段会让其余段落语义不完整，则它们属于同一模块
3. **尊重原文结构**：原文已有的分组（标题层级、分隔线、编号序列）是重要线索，优先沿用而非强行重组
4. **粒度适中**：模块太大等于没拆，太碎则增加认知负荷；以「一个模块可用一句话概括其职责」为粒度判断标准

## 常见切分维度（仅供启发，非必选清单）

- 角色与身份定义 vs 行为规则
- 通用准则 vs 场景化流程
- 静态知识（上下文、术语）vs 动态指令（操作步骤）
- 对外交互规范 vs 内部思维框架
- 工具/资源使用 vs 输出格式要求

具体采用哪些维度、拆出几个模块、如何命名，均由原文件内容决定——没有唯一正确答案，只有「对这份提示词最自然的切法」。
